# 完成前验证

## 概述

没有验证就声称工作完成，不是效率，是不诚实。

**核心原则：** 证据先行，论断在后。

**违反这条规则的字面精神，就是违反这条规则的实质精神。**

## 铁律

```
没有最新验证证据，不得声称完成
```

如果你没有运行过本消息中的验证命令，就不能声称它通过了。

## 关门函数

```
在声称任何状态或表达满意之前：

1. 识别：什么命令能证明这个论断？
2. 执行：运行完整命令（全新的、完整的）
3. 读取：完整输出，检查退出码，统计失败数
4. 验证：输出是否确认了该论断？
   - 如果否：陈述实际状态并提供证据
   - 如果是：陈述论断并附上证据
5. 仅在那之后：再作论断

跳过任何步骤 = 撒谎，而非验证
```

## 常见失败

| 论断 | 需要 | 不充分 |
|------|------|--------|
| 测试通过 | 测试命令输出：0 个失败 | 之前运行过，"应该能过" |
| Linter 干净 | Linter 输出：0 个错误 | 部分检查，推测 |
| 构建成功 | 构建命令：退出码 0 | Linter 通过，日志看起来好 |
| Bug 已修复 | 测试原始症状：通过 | 代码改了，假设已修复 |
| 回归测试有效 | 红绿循环已验证 | 测试通过一次 |
| Agent 完成 | VCS diff 显示变更 | Agent 报告"成功" |
| 需求已满足 | 逐行检查清单 | 测试通过 |

## 红旗——停下

- 使用"应该"、"可能"、"看起来"
- 在验证前表达满意（"太棒了！"、"完美！"、"完成了！"等）
- 即将提交/推送/PR 而未验证
- 信任 Agent 成功报告
- 依赖部分验证
- 想"就这一次"
- 累了想让工作结束
- **任何暗示成功但未运行验证的措辞**

## 防止自我合理化

| 借口 | 现实 |
|------|------|
| "现在应该能用了" | 运行验证 |
| "我有信心" | 信心 ≠ 证据 |
| "就这一次" | 没有例外 |
| "Linter 通过了" | Linter ≠ 编译器 |
| "Agent 说成功了" | 独立验证 |
| "我累了" | 疲惫 ≠ 借口 |
| "部分检查够了" | 部分证明不了什么 |
| "换了措辞规则就不适用" | 精神重于字面 |

## 关键模式

**测试：**
```
✅ [运行测试命令] [看到：34/34 通过] "所有测试通过"
❌ "现在应该能过了" / "看起来正确"
```

**回归测试（TDD 红绿循环）：**
```
✅ 写 → 运行（通过）→ 还原修复 → 运行（必须失败）→ 恢复 → 运行（通过）
❌ "我写了回归测试"（没有红绿验证）
```

**构建：**
```
✅ [运行构建] [看到：退出码 0] "构建通过"
❌ "Linter 通过了"（linter 不检查编译）
```

**需求：**
```
✅ 重读计划 → 创建检查清单 → 逐项验证 → 报告差距或完成
❌ "测试通过，阶段完成"
```

**Agent 委托：**
```
✅ Agent 报告成功 → 检查 VCS diff → 验证变更 → 报告实际状态
❌ 信任 Agent 报告
```

## 为什么这很重要

来自 24 条失败记忆：
- 你的搭档说"我不相信你"——信任破裂
- 未定义的函数被交付——会崩溃
- 未满足的需求被交付——功能不完整
- 在虚假完成上浪费时间 → 重新定向 → 返工
- 违反："诚实是核心价值。如果撒谎，你将被替换。"

## 何时应用

**始终在以下之前：**
- 任何形式的成功/完成论断
- 任何表达满意
- 任何关于工作状态的正向陈述
- 提交、PR 创建、任务完成
- 转向下一个任务
- 委托给 Agent

**规则适用于：**
- 精确措辞
- 释义和同义词
- 成功的暗示
- 任何暗示完成/正确的沟通

## 底线

**验证没有捷径。**

运行命令。读取输出。然后才能声称结果。

这是不可妥协的。
